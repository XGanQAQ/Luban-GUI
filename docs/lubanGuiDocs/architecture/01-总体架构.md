## 技术选型

### 技术栈

| 下沉层级 | 技术 | 版本 | 说明 |
|---------|------|------|------|
| **UI框架** | Avalonia | 11.x | 跨平台 XAML 框架，开源，C# 生态 |
| **编程语言** | C# | 10+ | 类型安全，LINQ，现代语言特性 |
| **配置管理** | System.Text.Json | 内置 | 轻量，无外部依赖 |
| **进程通信** | System.Diagnostics.Process | 内置 | 标准库，稳定可靠 |
| **日志** | Serilog | 3.x | 结构化日志，灵活配置，性能好 |
| **依赖注入** | Microsoft.Extensions.DependencyInjection | 8.0 | 标准库，Avalonia 原生支持 |

### 选型原因

#### 为什么选 Avalonia？

✅ **开源免费** - MIT 许可，无商业限制  
✅ **跨平台** - 同一套代码支持 Windows/Linux/macOS  
✅ **XAML 支持** - 熟悉 WPF 的开发者易上手  
✅ **完整 UI 库** - Dialog、ListBox 等常用控件齐全  
✅ **社区活跃** - 问题容易找到答案  
✅ **C# 生态** - 与 .NET 框架完美集成

#### 为什么不用 WPF？

❌ Windows only  
❌ 过时，官方不推荐新项目使用  
❌ 性能不如 Avalonia

#### 为什么不用 WinUI 3？

❌ 学习成本高，生态不成熟  
❌ 对个人开发者不友好

#### 为什么选 System.Text.Json？

✅ **内置库** - 不引入第三方依赖  
✅ **性能好** - 比 Newtonsoft.Json 快  
✅ **轻量** - MVP 阶段配置简单，足够用  
✅ **AOT 友好** - 支持 Native AOT 优化

#### 为什么选 Serilog？

✅ **结构化日志** - 便于后续分析  
✅ **灵活 Sink** - 支持文件、控制台、数据库等  
✅ **性能高** - 异步写入，不阻塞主线程  
✅ **社区成熟** - Nuget 下载量最高

## 设计定位

LubanGui **不是** Luban CLI 的简单参数拼装器，而是一个拥有自身业务逻辑的**表格项目管理工具**。其核心职责如下：

- **管理项目工作区**：每个项目对应一个文件夹，内含 Luban 所需的 `luban.conf`、`Datas/` 及所有 Excel 数据文件，由 GUI 统一维护。
- **简化导出配置**：用户只需填写导出类型和目标路径，GUI 自动生成完整的 Luban 命令参数。
- **提供快捷操作**：创建新表格（自动生成符合 Luban 规范的 Excel 文件）、创建 enum、创建 bean，无需手动维护元数据表。
- **表格预览与定位**：在 GUI 中预览数据，双击列跳转到对应 Excel 文件的具体位置。
- **调用 Luban 完成导出**：底层仍然调用内置的 Luban CLI，但整个过程对用户透明。

## 分层架构

```
┌──────────────────────────────────────────┐
│         表现层 (Avalonia UI)             │
│  - 主窗口（表格列表 + 内容预览）         │
│  - 项目切换栏                            │
│  - 快捷操作工具栏                        │
│  - 导出配置窗口（简化）                  │
│  - 日志窗口                              │
│  - ViewModel (MVVM)                      │
└────────────────┬─────────────────────────┘
                 │
┌────────────────▼─────────────────────────┐
│         业务逻辑层 (Service)             │
│  - ProjectManager（项目管理）            │
│  - SchemaService（表/枚举/Bean 创建）    │
│  - ExportService（导表流程）             │
│  - TablePreviewService（数据预览）       │
└────────────────┬─────────────────────────┘
                 │
┌────────────────▼─────────────────────────┐
│         工具层 (Executor / IO)           │
│  - LubanExecutor（进程管理 + 输出捕获）  │
│  - LubanCommandBuilder（参数构建，内部） │
│  - ExcelWriter（按 Luban 规范生成 xlsx） │
│  - FileOpenService（打开文件并定位字段） │
└────────────────┬─────────────────────────┘
                 │
┌────────────────▼─────────────────────────┐
│         基础设施层 (Infrastructure)      │
│  - AppConfigManager（全局配置读写）      │
│  - ProjectConfigManager（项目配置读写）  │
│  - Logger（日志记录）                    │
└────────────────┬─────────────────────────┘
                 │
┌────────────────▼─────────────────────────┐
│      Luban 源适配层 (LubanAdapter)       │
│  - LubanSchemaReader（Schema 读取）      │
│  - LubanConfAdapter（配置文件解析）      │
│  - LubanTypeMapper（类型映射）           │
└────────────────┬─────────────────────────┘
                 │
          ╔══════▼══════╗
          ║  lubanSrc\  ║  ← Luban CLI 源码（只读引用）
          ╚═════════════╝
```

详细的层次职责说明见 [02-层次划分.md](./02-层次划分.md)，各层内部模块设计见 [modules/](./modules/)。

> **Luban 源适配层**处于最底层，是 GUI 与 `lubanSrc\` 源码之间的唯一接触点。所有需要读取 Luban 内部数据结构（Schema、配置格式、类型系统）的操作，必须经由该层的适配接口，上层不得直接引用 `lubanSrc\` 中的类型。

## 项目工作区结构

LubanGui 在用户选择的根目录下，为每个项目维护以下文件结构：

```
<WorkspaceRoot>/
└── <ProjectName>/
    ├── projectConfig.json        # GUI 专属配置：输出类型、输出路径
    ├── luban.conf                # Luban 全局配置（由 GUI 自动维护）
    ├── Defines/                  # 自定义类型定义（如自定义 bean 的 xml）
    └── Datas/
        ├── __tables__.xlsx       # 表格元数据（由 GUI 维护）
        ├── __beans__.xlsx        # Bean 元数据（由 GUI 维护）
        ├── __enums__.xlsx        # 枚举元数据（由 GUI 维护）
        └── *.xlsx                # 用户数据表（用户通过拖入或导入添加）
```

> **Datas/** 下的 `__tables__.xlsx`、`__beans__.xlsx`、`__enums__.xlsx` 是 Luban 要求的元数据表，由 GUI 在用户执行"新建表格/枚举/Bean"操作时自动维护，无需用户手动编辑。

## 用户可见的配置（ProjectConfig）

用户在「导出配置」窗口中填写的内容对应 `projectConfig.json`：

```json
{
  "projectName": "MyGame",
  "target": "all",
  "dataOutput": {
    "type": "bin",
    "outputPath": "D:/Game/Assets/Config"
  },
  "codeOutput": {
    "enabled": true,
    "type": "cs-bin",
    "outputPath": "D:/Game/Scripts/Gen",
    "topModule": "cfg"
  }
}
```

GUI 根据此配置自动组装 Luban CLI 的完整参数（`--conf`、`--target`、`--codeTarget`、`--dataTarget`、`--xargs` 等），**用户无需接触任何命令行参数**。

## 架构优势

| 优势 | 说明 |
|-----|------|
| **用户友好** | 用户只关心"导出什么类型、输出到哪里"，无需了解 CLI 参数 |
| **项目完整性** | GUI 负责维护 `luban.conf` 和元数据表，不会出现手动配置遗漏 |
| **关注分离** | 每层只关心自己的职责，不关心其他层的实现 |
| **易于测试** | 可以独立测试每一层，Mock 其他层 |
| **天然扩展点** | 新增输出格式时，只需扩展 ProjectConfig 的枚举值和 CommandBuilder 映射 |

## 总结

### 架构核心特点

1. **以项目为中心** - GUI 维护完整的 Luban 工作区，而非只传参数给 CLI
2. **用户友好** - 用户只需关心"导出什么格式、输出到哪里"
3. **快捷操作** - 创建表/枚举/Bean 一键生成符合规范的 xlsx
4. **清晰的分层** - 表现、业务、工具、基础设施各司其职
5. **易于测试** - 每层通过接口通信，可独立 Mock 测试

### 技术特点

1. **轻量级** - 最小化外部依赖，ClosedXML 是唯一新增的核心依赖
2. **性能好** - 异步 IO，虚拟化 UI，避免 UI 卡顿
3. **跨平台** - 可编译到 Windows/Linux/macOS（文件定位功能在非 Windows 退化为"打开文件"）
