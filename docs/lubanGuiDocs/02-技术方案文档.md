## 技术选型

### 技术栈

| 下沉层级 | 技术 | 版本 | 说明 |
|---------|------|------|------|
| **UI框架** | Avalonia | 11.x | 跨平台 XAML 框架，开源，C# 生态 |
| **编程语言** | C# | 10+ | 类型安全，LINQ，现代语言特性 |
| **配置管理** | System.Text.Json | 内置 | 轻量，无外部依赖 |
| **进程通信** | System.Diagnostics.Process | 内置 | 标准库，稳定可靠 |
| **日志** | Serilog | 3.x | 结构化日志，灵活配置，性能好 |
| **依赖注入** | Microsoft.Extensions.DependencyInjection | 8.0 | 标准库，Avalonia 原生支持 |

### 选型原因

#### 为什么选 Avalonia？

✅ **开源免费** - MIT 许可，无商业限制  
✅ **跨平台** - 同一套代码支持 Windows/Linux/macOS  
✅ **XAML 支持** - 熟悉 WPF 的开发者易上手  
✅ **完整 UI 库** - Dialog、ListBox 等常用控件齐全  
✅ **社区活跃** - 问题容易找到答案  
✅ **C# 生态** - 与 .NET 框架完美集成

#### 为什么不用 WPF？

❌ Windows only  
❌ 过时，官方不推荐新项目使用  
❌ 性能不如 Avalonia

#### 为什么不用 WinUI 3？

❌ 学习成本高，生态不成熟  
❌ 对个人开发者不友好

#### 为什么选 System.Text.Json？

✅ **内置库** - 不引入第三方依赖  
✅ **性能好** - 比 Newtonsoft.Json 快  
✅ **轻量** - MVP 阶段配置简单，足够用  
✅ **AOT 友好** - 支持 Native AOT 优化

#### 为什么选 Serilog？

✅ **结构化日志** - 便于后续分析  
✅ **灵活 Sink** - 支持文件、控制台、数据库等  
✅ **性能高** - 异步写入，不阻塞主线程  
✅ **社区成熟** - Nuget 下载量最高

## 整体架构

### 设计定位

LubanGui **不是** Luban CLI 的简单参数拼装器，而是一个拥有自身业务逻辑的**表格项目管理工具**。其核心职责如下：

- **管理项目工作区**：每个项目对应一个文件夹，内含 Luban 所需的 `luban.conf`、`Datas/` 及所有 Excel 数据文件，由 GUI 统一维护。
- **简化导出配置**：用户只需填写导出类型和目标路径，GUI 自动生成完整的 Luban 命令参数。
- **提供快捷操作**：创建新表格（自动生成符合 Luban 规范的 Excel 文件）、创建 enum、创建 bean，无需手动维护元数据表。
- **表格预览与定位**：在 GUI 中预览数据，双击列跳转到对应 Excel 文件的具体位置。
- **调用 Luban 完成导出**：底层仍然调用内置的 Luban CLI，但整个过程对用户透明。

### 项目工作区结构

LubanGui 在用户选择的根目录下，为每个项目维护以下文件结构：

```
<WorkspaceRoot>/
└── <ProjectName>/
    ├── projectConfig.json        # GUI 专属配置：输出类型、输出路径
    ├── luban.conf                # Luban 全局配置（由 GUI 自动维护）
    ├── Defines/                  # 自定义类型定义（如自定义 bean 的 xml）
    └── Datas/
        ├── __tables__.xlsx       # 表格元数据（由 GUI 维护）
        ├── __beans__.xlsx        # Bean 元数据（由 GUI 维护）
        ├── __enums__.xlsx        # 枚举元数据（由 GUI 维护）
        └── *.xlsx                # 用户数据表（用户通过拖入或导入添加）
```

> **Datas/** 下的 `__tables__.xlsx`、`__beans__.xlsx`、`__enums__.xlsx` 是 Luban 要求的元数据表，由 GUI 在用户执行"新建表格/枚举/Bean"操作时自动维护，无需用户手动编辑。

### 用户可见的配置（ProjectConfig）

用户在「导出配置」窗口中填写的内容对应 `projectConfig.json`：

```json
{
  "projectName": "MyGame",
  "target": "all",
  "dataOutput": {
    "type": "bin",
    "outputPath": "D:/Game/Assets/Config"
  },
  "codeOutput": {
    "enabled": true,
    "type": "cs-bin",
    "outputPath": "D:/Game/Scripts/Gen",
    "topModule": "cfg"
  }
}
```

GUI 根据此配置自动组装 Luban CLI 的完整参数（`--conf`、`--target`、`--codeTarget`、`--dataTarget`、`--xargs` 等），**用户无需接触任何命令行参数**。

### 架构优势

| 优势 | 说明 |
|-----|------|
| **用户友好** | 用户只关心"导出什么类型、输出到哪里"，无需了解 CLI 参数 |
| **项目完整性** | GUI 负责维护 `luban.conf` 和元数据表，不会出现手动配置遗漏 |
| **关注分离** | 每层只关心自己的职责，不关心其他层的实现 |
| **易于测试** | 可以独立测试每一层，Mock 其他层 |
| **天然扩展点** | 新增输出格式时，只需扩展 ProjectConfig 的枚举值和 CommandBuilder 映射 |

### 模块2: 业务逻辑层 (Service)

#### ProjectManager

**职责**: 管理多个项目的创建、切换、加载

**接口**:
```csharp
public interface IProjectManager
{
    // 获取所有已注册的项目
    IReadOnlyList<ProjectInfo> Projects { get; }

    // 当前打开的项目
    ProjectInfo? CurrentProject { get; }

    // 创建新项目（在指定路径下初始化工作区）
    Task<ProjectInfo> CreateProjectAsync(string name, string workspacePath);

    // 打开已有项目文件夹
    Task<ProjectInfo> OpenProjectAsync(string workspacePath);

    // 切换当前项目
    Task SwitchProjectAsync(string projectName);

    // 删除项目记录（不删除文件）
    void RemoveProject(string projectName);
}
```

**工作区初始化流程（CreateProjectAsync）**:
```
1. 在 workspacePath 下创建 <name>/ 目录
2. 生成初始 luban.conf（包含默认 groups、schemaFiles、targets）
3. 生成初始 Datas/__tables__.xlsx、__beans__.xlsx、__enums__.xlsx
4. 生成初始 projectConfig.json（空的输出配置）
5. 将项目信息写入全局 appConfig.json
```

#### SchemaService

**职责**: 维护 Luban 元数据表（`__tables__`、`__beans__`、`__enums__`），提供创建新表/枚举/Bean 的快捷操作

**接口**:
```csharp
public interface ISchemaService
{
    // 读取当前项目下所有已注册的表格
    Task<IReadOnlyList<TableMeta>> GetTablesAsync();

    // 创建新数据表：在 Datas/ 下生成符合 Luban 规范的 xlsx，并写入 __tables__.xlsx
    Task<TableMeta> CreateTableAsync(NewTableOptions options);

    // 注册已有 xlsx 文件为数据表（用户拖入文件后调用）
    Task<TableMeta> ImportTableAsync(string xlsxPath);

    // 创建新枚举：写入 __enums__.xlsx
    Task CreateEnumAsync(NewEnumOptions options);

    // 创建新 Bean：写入 __beans__.xlsx
    Task CreateBeanAsync(NewBeanOptions options);

    // 从 __tables__.xlsx 同步删除一条表格记录
    Task RemoveTableAsync(string tableName);
}
```

**新建表格时生成的 xlsx 格式**（符合 Luban 约定）:
```
行1: ##  id      name     value    desc
行2: ##  int     string   int      string
行3: ##          (group)
行4:     1       示例     100      示例说明
```

#### ExportService

**职责**: 根据 `ProjectConfig` 编排完整的导表流程

**接口**:
```csharp
public interface IExportService
{
    Task<ExportResult> ExportAsync(
        ProjectConfig config,
        IProgress<string> logProgress,
        CancellationToken cancellationToken = default);

    ValidationResult ValidateConfig(ProjectConfig config);
}
```

**流程**:
```
1. 验证 ProjectConfig（输出路径非空，输出类型有效）
   ↓
2. 确保输出目录存在（CreateDirectory）
   ↓
3. 将 ProjectConfig 映射为 LubanRunArgs（内部，用户不可见）
   ↓
4. 调用 LubanExecutor 执行 Luban CLI
   ↓
5. 保存 ProjectConfig 到 projectConfig.json
   ↓
6. 返回结果
```

**异常处理**:
- `ConfigInvalidException` → 显示配置错误
- `ProcessException` → 显示 Luban 运行时错误
- `OperationCanceledException` → 显示"已取消"
- 通用 `Exception` → 显示"未知错误"

**读写操作**:
```csharp
public async Task<ProjectConfig> LoadAsync(string projectDir)
{
    var path = Path.Combine(projectDir, "projectConfig.json");
    if (!File.Exists(path))
        return new ProjectConfig();
    using var stream = File.OpenRead(path);
    return await JsonSerializer.DeserializeAsync<ProjectConfig>(stream);
}

public async Task SaveAsync(string projectDir, ProjectConfig config)
{
    var path = Path.Combine(projectDir, "projectConfig.json");
    using var stream = File.Create(path);
    await JsonSerializer.SerializeAsync(stream, config, _jsonOptions);
}
```

## 关键技术方案

### 方案1: 线程安全的 UI 更新

**问题**: LubanExecutor 在后台线程读取进程输出，如何安全地更新 UI？

**方案**: 使用 `IProgress<T>`

```csharp
// 后台线程中
logProgress?.Report(logLine);  // 线程安全，自动切换到 UI 线程

// ViewModel 中
var progress = new Progress<string>(line =>
{
    // 自动在 UI 线程执行
    LogItems.Add(line);
});
```

### 方案2: 实时日志显示

**问题**: 如果等到进程完成再显示日志，用户要等很久

**方案**: 逐行实时推送

```
进程输出 → StreamReader → 逐行读 → IProgress.Report() → UI 更新
     (异步)         (逐行)         (实时)          (自动线程切换)
```

**性能优化**:
- 使用虚拟化列表（ListBox.VirtualizationMode = Item）
- 限制日志条数（保留最近 10000 行）
- 异步写入磁盘

### 方案3: 进程间的输出捕获

**问题**: Luban 输出到 stdout/stderr，我们如何获取？

**方案**: 重定向并异步读取

```
Luban 进程
  ├─ stdout → RedirectStandardOutput=true → stream → ReadLineAsync()
  └─ stderr → RedirectStandardError=true → stream → ReadLineAsync()
```

**关键设置**:
```csharp
var psi = new ProcessStartInfo
{
    RedirectStandardOutput = true,      // ✓ 重定向 stdout
    RedirectStandardError = true,       // ✓ 重定向 stderr
    UseShellExecute = false,            // ✓ 不通过 cmd.exe
    CreateNoWindow = true,              // ✓ 不创建窗口
    StandardOutputEncoding = Encoding.UTF8  // ✓ UTF-8 编码
};
```

### 方案4: 避免进程通信死锁

**问题**: 进程缓冲区有限，如果只读一个流，当缓冲区满时会死锁

**解决**: 同时读取两个流

```csharp
// ✗ 死锁：先读完 stdout，再读 stderr，缓冲区满时卡住
string out = process.StandardOutput.ReadToEnd();
string err = process.StandardError.ReadToEnd();

// ✓ 正确：同时读两个流
Task<string> outTask = ReadStreamAsync(process.StandardOutput);
Task<string> errTask = ReadStreamAsync(process.StandardError);
await Task.WhenAll(outTask, errTask);
```

## 总结

### 架构核心特点

1. **以项目为中心** - GUI 维护完整的 Luban 工作区，而非只传参数给 CLI
2. **用户友好** - 用户只需关心"导出什么格式、输出到哪里"
3. **快捷操作** - 创建表/枚举/Bean 一键生成符合规范的 xlsx
4. **清晰的分层** - 表现、业务、工具、基础设施各司其职
5. **易于测试** - 每层通过接口通信，可独立 Mock 测试

### 技术特点

1. **轻量级** - 最小化外部依赖，ClosedXML 是唯一新增的核心依赖
2. **性能好** - 异步 IO，虚拟化 UI，避免 UI 卡顿
3. **跨平台** - 可编译到 Windows/Linux/macOS（文件定位功能在非 Windows 退化为"打开文件"）
