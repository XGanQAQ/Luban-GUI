## 一、文档目的

本文档面向**策划和非程序人员**，帮助理解"导出配置"窗口中各字段的含义与选择依据，以便在无需了解 Luban CLI 底层参数的前提下，独立完成导出配置的填写。

> 本文只讨论**"需要配置什么"和"为什么这样配置"**，不涉及任何代码实现。

---

## 二、Luban 导出机制概述

Luban 在执行时，会将 Excel 配置表中的数据按照指定的格式输出为两类产物：

| 产物 | 说明 | 使用方 |
|------|------|--------|
| **数据文件** | 运行时加载的配置数据，如 `.bin`、`.json`、`.lua` 文件 | 程序在游戏/服务运行时读取 |
| **代码文件** | 读取上述数据文件所需的类型定义代码，如 C# / TypeScript / Java 类 | 程序员在项目中引入，用来访问配置字段 |

Luban-GUI 将这一过程简化为一个配置窗口。策划只需填写"输出什么格式"和"输出到哪里"，GUI 会自动将这些选择转化为 Luban CLI 的底层参数，策划**无需了解任何命令行参数**。

## 三、内置 Luban 与 DLL 路径

Luban 本身是一个 .NET 命令行工具，GUI 调用它时需要知道 `Luban.dll` 的所在路径。

### 内置 Luban

**Luban-GUI 在安装包中内置了一份已构建好的 Luban 工具**，存放于应用程序目录下的 `luban\` 子目录（即 `<应用程序目录>\luban\Luban.dll`）。

**在用户未手动指定 DLL 路径的情况下，GUI 默认自动使用内置版本**，无需任何额外配置，开箱即用。

### 自定义 DLL 路径

如有需要（例如项目要求使用特定版本的 Luban，或需要使用内置版本以外的自定义构建），可在应用设置中手动指定 `Luban.dll` 的完整路径，GUI 将优先使用指定路径。

| 情况 | DLL 来源 |
|------|---------|
| 未指定路径（默认） | 使用 GUI 内置的 `luban\Luban.dll` |
| 已手动指定路径 | 使用用户指定的 `Luban.dll` |

> **策划日常使用无需关心此设置**，保持默认即可。仅当程序员要求切换特定 Luban 版本时才需要调整。

## 四、核心概念解释

### 数据导出（Data Target）

"数据导出"决定 Excel 配置表的内容以**什么格式保存为数据文件**，供程序在运行时加载。

不同格式在文件大小、可读性、加载效率上有所不同：

| 格式 | 特点 | 推荐场景 |
|------|------|---------|
| `bin` | 体积最小、加载最快，但不可直接阅读 | 正式发布版本 |
| `json` | 纯文本，可用记事本打开查看，体积较大 | 调试、脚本语言项目 |
| `lua` | Lua 语言原生格式 | 使用 Lua 的游戏项目 |
| 其他（xml、yaml、bson 等） | 特殊场景使用 | 由程序员决定 |

> **策划日常导表通常只需了解 `bin` 和 `json` 两种。** 项目初期一般由程序员决定使用哪种，之后策划只需沿用即可。

---

### 代码导出（Code Target）

"代码导出"决定是否**同时生成配套的代码文件**（访问配置数据的类型定义）。

- **不生成代码**：只输出数据文件，适合配置表结构稳定、代码已经存在的情况（例如只有少量数据变更）。
- **生成代码**：同时输出代码文件，适合新增字段、修改表结构后，需要让程序员获得最新类型定义的情况。

代码类型（如 `cs-bin`、`typescript-json`）需要与项目使用的编程语言和数据格式匹配，通常**由程序员决定并告知策划**，策划只需照填即可。

---

### 导出目标（Target / Group）

"导出目标"决定**哪些表格和字段参与本次导出**。

Luban 支持将配置表按"组"划分（通常是客户端 `client`、服务器 `server`），不同目标只导出各自需要的数据，避免泄露对方不应看到的配置内容。

| 目标名 | 含义 |
|--------|------|
| `all` | 导出所有分组（客户端 + 服务器）的数据，通常用于内部开发环境 |
| `client` | 只导出客户端所需数据 |
| `server` | 只导出服务器所需数据 |

> **目标名称由项目的 `luban.conf` 定义**，GUI 会自动读取可用的目标并显示在下拉框中，策划无需手动输入。不同项目可能有自定义的目标名称。

## 五、导出配置字段详解

「导出配置」窗口（通过「操作 > 导出配置...」打开）包含以下字段：

```
┌──────────────────── 导出配置 ──────────────────────────────────────┐
│                                                                    │
│  项目: MyGame                                                      │
│                                                                    │
│  ▾ 数据导出 ─────────────────────────────────────────────────     │
│    导出类型:    [bin               ▾]                              │
│    输出路径:    [__________________________________] [📂 浏览]     │
│                                                                    │
│  ▾ 代码导出 ─────────────────────────────────────────────────     │
│    ☑ 生成代码                                                      │
│    代码类型:    [cs-bin            ▾]                              │
│    输出路径:    [__________________________________] [📂 浏览]     │
│    顶层命名空间: [cfg_________________________________]            │
│                                                                    │
│  ▾ 导出目标 ─────────────────────────────────────────────────     │
│    目标:        [all               ▾]                              │
│                  说明: all = 导出 client + server 两端数据         │
│                                                                    │
│                                      [取消]  [保存并关闭]          │
└────────────────────────────────────────────────────────────────────┘
```

### 数据导出区字段

| 字段 | 必填 | 说明 | 示例值 |
|------|------|------|--------|
| **导出类型** | ✅ | 数据文件的序列化格式（见第五节） | `bin`、`json` |
| **输出路径** | ✅ | 导出数据文件存放的文件夹路径 | `D:\Game\Assets\Config` |

### 代码导出区字段

| 字段 | 必填 | 说明 | 示例值 |
|------|------|------|--------|
| **生成代码** | — | 勾选后才会导出代码文件；不勾选则仅导出数据 | ☑ / ☐ |
| **代码类型** | 勾选时必填 | 代码语言及读取格式（见第六节） | `cs-bin`、`typescript-json` |
| **输出路径** | 勾选时必填 | 生成代码文件存放的文件夹路径 | `D:\Game\Scripts\Gen` |
| **顶层命名空间** | 否 | 生成代码所在的顶层模块/命名空间名称 | `cfg` |

> "生成代码"取消勾选时，代码类型、代码输出路径、顶层命名空间均自动置灰，无需填写。

### 导出目标区字段

| 字段 | 必填 | 说明 | 示例值 |
|------|------|------|--------|
| **目标** | ✅ | 本次导出所包含的分组范围，从 `luban.conf` 动态读取 | `all`、`client`、`server` |

## 六、数据格式选项说明

GUI 在"导出类型"下拉框中提供以下常用选项：

| 选项 | 文件特征 | 推荐使用场景 |
|------|---------|-------------|
| `bin` | 二进制，体积小，不可阅读 | Unity/客户端正式发布；服务器高性能场景 |
| `json` | 文本格式，可直接查看内容 | 脚本语言项目（Python/JS）；调试验证 |
| `json2` | 与 json 类似，但 map 输出格式不同 | 需要 `{"key":"value"}` 格式的 json 场景 |
| `lua` | Lua 原生表格格式 | 使用 Lua 作为逻辑语言的游戏（Cocos 等） |
| `xml` | XML 格式文本 | 特定平台或工具链有 XML 要求时 |
| `yaml` | YAML 格式文本 | 偏好 YAML 配置风格的服务器项目 |
| `msgpack` | MessagePack 二进制格式 | 对体积敏感、跨语言通信场景 |

> **不确定选哪个？** 直接询问项目程序员，通常项目初期已有约定。

---

## 七、代码类型选项说明

代码类型由**使用的编程语言**和**配套读取的数据格式**共同决定，命名规律为 `{语言}-{数据格式}`。

### 常用组合

| 代码类型 | 语言 | 配套数据格式 | 适用平台/框架 |
|---------|------|------------|-------------|
| `cs-bin` | C# | bin | Unity（高性能客户端）/ dotnet 服务器 |
| `cs-simple-json` | C# | json | Unity 客户端（轻量 JSON） |
| `cs-dotnet-json` | C# | json | dotnet core 服务器 |
| `cs-newtonsoft-json` | C# | json | 使用 Newtonsoft.Json 的 C# 项目 |
| `typescript-json` | TypeScript | json | Cocos Creator / Web 前端项目 |
| `typescript-bin` | TypeScript | bin | TypeScript 高性能场景 |
| `java-bin` | Java | bin | Android / Java 服务器 |
| `java-json` | Java | json | Java 服务器（Gson 库） |
| `go-bin` | Go | bin | Go 语言服务器 |
| `go-json` | Go | json | Go 语言服务器（JSON 格式） |
| `python-json` | Python | json | Python 脚本/工具 |
| `lua-bin` | Lua | bin | Lua + bin 格式 |
| `lua-lua` | Lua | lua | Lua + Lua 格式 |

> **注意**：代码类型必须与数据格式匹配（例如 `cs-bin` 只能读取 `bin` 格式数据文件），否则程序运行时会加载失败。

### 顶层命名空间（TopModule）

- 生成代码时，所有配置相关的类型会被放在指定的命名空间下。
- 例如填写 `cfg`，则 C# 代码中访问配置时写 `cfg.Tables`、`cfg.TbItem`。
- 留空时代码不会添加额外的命名空间前缀。
- **通常由程序员决定，策划直接复制程序员告知的值即可**（一般为 `cfg`）。

## 八、典型配置场景

### 场景 A：Unity + C# 客户端（正式发布）

> 目标：生成高效的二进制数据文件和 C# 读取代码，用于 Unity 客户端。

| 字段 | 填写值 |
|------|--------|
| 数据导出类型 | `bin` |
| 数据输出路径 | Unity 项目内的 `Assets/Config/` 或 `StreamingAssets/Config/` |
| 生成代码 | ☑ 勾选 |
| 代码类型 | `cs-bin` |
| 代码输出路径 | Unity 项目内的 `Assets/Scripts/Gen/` |
| 顶层命名空间 | `cfg` |
| 导出目标 | `client` |

---

### 场景 B：Unity + C# 客户端（开发调试阶段）

> 目标：使用 JSON 格式便于直接查看文件内容，快速排查数据问题。

| 字段 | 填写值 |
|------|--------|
| 数据导出类型 | `json` |
| 数据输出路径 | Unity 项目内的 `Assets/Config/` |
| 生成代码 | ☑ 勾选 |
| 代码类型 | `cs-simple-json` |
| 代码输出路径 | Unity 项目内的 `Assets/Scripts/Gen/` |
| 顶层命名空间 | `cfg` |
| 导出目标 | `all` |

---

### 场景 C：Cocos Creator + TypeScript

> 目标：生成 JSON 格式数据和 TypeScript 读取代码。

| 字段 | 填写值 |
|------|--------|
| 数据导出类型 | `json` |
| 数据输出路径 | Cocos 项目内的 `assets/resources/config/` |
| 生成代码 | ☑ 勾选 |
| 代码类型 | `typescript-json` |
| 代码输出路径 | Cocos 项目内的 `assets/scripts/gen/` |
| 顶层命名空间 | `cfg` |
| 导出目标 | `client` |

---

### 场景 D：仅导出数据（不重新生成代码）

> 适用于：表格结构未变，只有数据内容修改，程序侧代码无需更新。

| 字段 | 填写值 |
|------|--------|
| 数据导出类型 | `bin`（或项目约定格式） |
| 数据输出路径 | 项目数据目录 |
| 生成代码 | ☐ 不勾选 |
| 导出目标 | `all`（或具体目标） |

---

### 场景 E：仅校验配置，不生成任何文件

> 适用于：策划在提交前想检查配置表是否有错误，但不想输出实际文件。

使用菜单「操作 > 校验配置」（`F6`），GUI 会在不输出任何文件的情况下完成全量校验，并在日志窗口中显示错误信息。

> 此操作**无需填写输出路径**，可以在路径尚未确定时提前检查表格格式。

## 九、代码格式与数据格式的配套关系

**代码类型和数据格式必须匹配**，否则运行时会报加载错误。下表列出合法的搭配：

| 数据格式 | 可搭配的代码类型（示例） |
|---------|------------------------|
| `bin` | `cs-bin`、`lua-bin`、`java-bin`、`go-bin`、`typescript-bin`、`javascript-bin`、`rust-bin`、`cpp-sharedptr-bin`、`cpp-rawptr-bin` |
| `json` / `json2` | `cs-simple-json`、`cs-dotnet-json`、`cs-newtonsoft-json`、`typescript-json`、`java-json`、`go-json`、`python-json`、`php-json`、`dart-json`、`gdscript-json`、`rust-json` |
| `lua` | `lua-lua` |
| `protobuf2-bin` / `protobuf3-bin` | `cs-protobuf2`、`cs-protobuf3`、`typescript-protobuf` |
| `flatbuffers-json` | `flatbuffers` |

> **简化记忆**：代码类型名称中后半部分（`-bin`、`-json`、`-lua`）即表示它能读取的数据格式。

## 十、注意事项与常见问题

### ① 输出路径是否需要提前创建？

不需要。GUI 在执行导表前会自动检查并创建不存在的输出目录。

---

### ② 数据文件会覆盖旧文件吗？

是的。每次导表会重新生成并覆盖同名文件。如需保留历史版本，建议提前做好版本管理（Git 等）。

---

### ③ 导出目标下拉框为空怎么办？

下拉框内容从当前项目的 `luban.conf` 中读取。若为空，可能是：
- 项目未正确初始化，`luban.conf` 不存在或路径有误
- `luban.conf` 中的 `targets` 字段为空

可联系程序员确认 `luban.conf` 文件内容。

---

### ④ 我能同时导出客户端和服务器两套数据吗？

v1.0 的"导出配置"窗口每次只能配置**一组**数据导出目标。  
- 若使用目标 `all`，则会将客户端和服务器数据合并导出到同一个目录。
- 若需要分别输出到不同目录，计划在 **v1.1 通过「多导出配置预设」功能支持**（见第十节）。

---

### ⑤ "顶层命名空间"填错了会怎样？

顶层命名空间只影响生成的**代码文件内容**，不影响数据文件。若填错，程序员在引用生成代码时可能会遇到找不到类型的编译错误，重新导出即可修复。

---

### ⑥ 导表失败了如何排查？

1. 查看**日志窗口**（`Ctrl+L`）中的 `[ERROR]` 行，Luban 会指出问题所在的表格和行号。
2. 如果提示"找不到文件"，检查输出路径是否有拼写错误，或者磁盘空间是否不足。
3. 如果提示"字段类型不匹配"，双击主窗口表格列表中的问题表格列标题，GUI 会自动定位到 Excel 文件的对应列。

## 十一、后续版本规划

### v1.1：多导出配置预设

同一项目下可保存多套导出配置（如"出客户端 bin"和"出服务器 json"），在项目切换栏中快速切换，一键导出对应目标。

**场景示例**：

| 预设名 | 数据格式 | 代码类型 | 目标 |
|--------|---------|---------|------|
| 客户端发布 | `bin` | `cs-bin` | `client` |
| 服务器发布 | `json` | `cs-dotnet-json` | `server` |
| 全量调试 | `json` | `cs-simple-json` | `all` |

---

### v1.2：导表历史与配置回溯

保存每次导表的配置快照（时间、预设名、结果、Luban 版本），支持查看历史日志，便于追溯"这次导出用的是什么配置"。
